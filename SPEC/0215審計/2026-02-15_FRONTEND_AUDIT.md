# 前端程式品質審查報告 (2026-02-15)

## 📑 審查概述
對「添心生產力助手」客戶端（Electron 專案）進行源碼掃描，重點評估資訊安全、本地數據持久化穩定性與系統資源佔用。

---

## 🔍 已識別之缺陷與改善方案 (Identified Risks & Remediation)

### 1. 核心安全漏洞：硬編碼管理員密碼 (Hard-coded Credentials)
- **位置**：`config.js` -> `ADMIN_PASSWORD` (Line 13)
- **目前運作方式 (白話說明)**：
  程式碼裡面像寫便利貼一樣，直接把最高權限的管理員密碼 `Tx2649819` 寫在裡面。
- **風險**：任何懂一點電腦的人，只要把程式打開來看（反編譯），就能看到這組密碼，然後就能隨意進入管理後台亂改設定。
- **改善方案 A：環境變數 (Environment Variables) [推薦]**
  - **作法**：使用 `dotenv` 套件，將密碼移至 `.env` 檔案，並在打包時透過 `electron-builder` 的 `extraResources` 或加密配置注入。
- **改善方案 B：雜湊驗證 (Hash Verification)**
  - **作法**：原始碼中僅儲存密碼的 Hash (e.g., bcrypt/Argon2)，驗證時將輸入值 Hash 後比對。雖然不能防止暴力破解，但避免了明文洩漏。

### 2. 資料遺失風險：全記憶體資料庫運作 (In-Memory DB Persistence)
- **位置**：`storage.js` -> `init`, `saveToFile`
- **目前運作方式 (白話說明)**：
  程式運作時，是把所有的資料（員工活動紀錄）都記在隨機存取記憶體 (RAM) 裡，就像我們在記事本打字但沒按存檔一樣。系統規定每 5 秒鐘才會按一次「存檔」。
- **風險**：如果在剛好第 4.9 秒的時候電腦當機或斷電，這將近 5 秒內的資料就會憑空消失。而且當資料越來越多，每 5 秒一次的存檔動作會讓電腦頓一下。
- **改善方案 A：SQLite WAL 模式 [推薦]**
  - **作法**：改用 `better-sqlite3` 或 `sqlite3` 原生模組，開啟 Write-Ahead Logging (WAL) 模式。這能確保每一筆寫入都即時持久化，且不會阻塞主執行緒。
- **改善方案 B：增量日誌 (Append-only Log)**
  - **作法**：維持記憶體資料庫的高速讀取，但寫入時同時 Append 到一個 `journal.jsonl` 檔案。崩潰重啟時讀取該 Log 恢復資料。

### 3. 性能與資源洩漏：視窗管理不當 (Resource Leak in UI)
- **位置**：`monitor.js` -> `showToast`
- **目前運作方式 (白話說明)**：
  每當要跳出一個警示視窗（例如「你閒置太久了」），程式就會**新買一個視窗**，用完後再丟掉。
- **風險**：如果在很短時間內連續跳出很多警示，電腦的記憶體會被這些「新視窗」塞滿，導致程式越來越慢，甚至讓電腦卡死。
- **改善方案 A：單例模式 (Singleton Pattern) [推薦]**
  - **作法**：全域只維護一個 `ToastWindow` 實例。需要顯示時呼叫 `.show()` 並更新內容，不需要時呼叫 `.hide()` 而非 `.close()` / `.destroy()`，避免反覆的 GC 與物件創建開銷。

### 4. 數據同步脆弱性 (Fragile Data Sync)
- **位置**：`reporter.js` -> `reportNow`
- **目前運作方式 (白話說明)**：
  當要回傳資料給伺服器時，程式會試著**一次把所有累積的資料丟過去**。如果傳送失敗（例如網路斷線），它就放棄了，下次再全部重丟一次。
- **風險**：如果公司網路不穩定，或者累積的資料太多（例如累積了一整天），資料包會大到傳不出去，結果就是資料永遠卡在員工的電腦裡，主管完全看不到報表。
- **改善方案 A：持久化隊列 (Persistent Queue) [推薦]**
  - **作法**：建立一個 `upload_queue` 資料表。所有要上傳的資料先寫入 Queue。每一筆上傳成功後才從 Queue 刪除 (`ACK` 機制)。背景 Worker 定期消耗 Queue。
- **改善方案 B：分塊上傳 (Chunked Upload)**
  - **作法**：若資料陣列長度 > 100，自動切分為多個 Request 依序發送。

### 5. 敏感資訊明文儲存 (Insecure Storage)
- **位置**：`config.js` 與 `config.json` (Disk)
- **目前運作方式 (白話說明)**：
  連線到後端所需的鑰匙 (API Key)，程式是直接用純文字檔案 (`config.json`) 存在員工電腦的硬碟裡。
- **風險**：這就像把家門鑰匙放在門口地墊下一樣。如果這台電腦中了病毒，病毒可以直接打開這個文字檔，偷走鑰匙，然後假冒員工身分竊取公司資料。
- **改善方案 A：系統金鑰庫 (System Keychain) [推薦]**
  - **作法**：使用 `keytar` 庫，將 API Key 存入 Windows Credential Manager / macOS Keychain。這是作業系統級別的安全儲存。
- **改善方案 B：AES 加密儲存**
  - **作法**：在寫入 JSON 前使用 AES-256 加密，金鑰混淆藏於程式碼中 (Obfuscation)。雖非絕對安全，但可防制一般腳本小子。

### 6. 分類邏輯不一致與硬編碼 (Static Classification)
- **位置**：`monitor.js` (午休/閒置邏輯)
- **目前運作方式 (白話說明)**：
  午休時間是 12:00 到 13:00，這規則是**寫死在程式碼**裡的。
- **風險**：如果公司政策改變，或者有部門是 12:30 才午休，這個程式就會誤判員工在「摸魚」，且無法透過後台設定修改，必須請工程師重新改程式並發布更新。
- **改善方案 A：遠端動態配置 (Remote Config) [推薦]**
  - **作法**：APP 啟動時從後端 API 拉取 `policy.json` (含午休時間、閒置閾值)，並快取於本地。
- **改善方案 B：本地設定檔**
  - **作法**：將這些變數移至 `config.json`，允許透過管理介面修改。

### 7. 低效的分類比對演算法 (Inefficient Matching Algorithm)
- **位置**：`classifier.js` -> `matchList`, `matchKeywords`
- **目前運作方式 (白話說明)**：
  為了判斷員工正在看的網頁是不是「工作相關」，程式會拿標題去跟幾百個關鍵字**一個一個比對**。
- **風險**：當關鍵字變多（例如有 1000 個），每 3 秒鐘就要比對 1000 次，這會讓員工的電腦變得很慢，CPU 風扇狂轉，被員工抱怨電腦變卡。
- **改善方案 A：Aho-Corasick 演算法 [推薦]**
  - **作法**：使用多模式字串匹配演算法 (如 `aho-corasick-node`)，在 O(n) 時間內完成數千個關鍵字的同步比對。
- **改善方案 B：正則表達式優化**
  - **作法**：將所有關鍵字預編譯為一個巨大的 Regex (`/keyword1|keyword2|.../`)，利用 V8 引擎底層的優化來加速。

### 8. 脆弱的瀏覽器標題抓取 (Brittle Title Scraping)
- **位置**：`classifier.js` -> `extractSiteName`
- **目前運作方式 (白話說明)**：
  程式是靠判斷視窗標題的最後幾個字（例如 "... - Google Chrome"）來知道這是什麼瀏覽器。
- **風險**：如果不小心瀏覽器更新了，或是員工把電腦語言改成英文（變成 "... - Google Chrome"），程式就會看不懂，導致抓到的網頁標題是一堆亂碼，報表也會跟著爛掉。
- **改善方案 A：無障礙 API (Accessibility API) [推薦]**
  - **作法**：不依賴視窗標題，改用 OS 的 Accessibility API (如 Windows UI Automation) 直接讀取瀏覽器網址列 (Address Bar) 的 URL 進行網域分類，更為精準。

### 9. 缺乏 API 連線逾時處理機制 (Missing Robust Timeout Handling)
- **位置**：`reporter.js` -> `sendReport`
- **目前運作方式 (白話說明)**：
  程式設定如果 30 秒沒傳成功就當作失敗。除此之外，遇到網路瞬斷或 DNS 解析失敗，它就直接報錯。
- **風險**：在辦公室 Wi-Fi 訊號不好的地方，程式會一直跳出「連線錯誤」的內部訊息，消耗電腦資源，且無法自動切換備援機制。
- **改善方案 A：指數退避重試 (Exponential Backoff) [推薦]**
  - **作法**：建立 `fetchWithRetry` 封裝，遭遇網路錯誤時等待 1s, 2s, 4s, 8s 後重試，最多 5 次。避免瞬間流量風暴並增加成功率。

### 10. 運行時崩潰錯誤 (Runtime Crash / Bug)
- **位置**：`reminderService.js` -> `start` (Line 282)
- **狀況**：程式碼嘗試對 `const triggerTime` 進行重新賦值 (`triggerTime = new Date(...)`)。
- **風險**：這是 JavaScript 的語法錯誤。當使用者有「未打卡」的提醒需要延後觸發時，程式會直接崩潰 (Crash) 並強制關閉，導致提醒功能完全失效。
- **改善方案 A：修正變數宣告 [立即]**
  - **作法**：將 Line 269 的 `const triggerTime` 改為 `let triggerTime`。

## ⚖️ 修復優先級建議 (Priority Suggestions)

根據風險嚴重性與修復成本，建議的處理順序如下：

| 優先級 | 項目 (Issue) | 風險等級 | 建議行動 |
| :--- | :--- | :--- | :--- |
| **P0** | **管理員密碼硬編碼** | Critical | 立即移除原始碼中的密碼，改用環境變數或加密雜湊驗證。 |
| **P0** | **敏感資訊明文儲存** | High | 使用 `keytar` 或系統級憑證管理庫來儲存 API Key，而非明文 JSON。 |
| **P1** | **全記憶體資料庫風險** | High | 改用 SQLite 的 WAL 模式或定期增量備份，避免單點故障導致資料全毀。 |
| **P1** | **資料同步脆弱性** | High | 實作本地 Queue 機制與失敗重試 (Exponential Backoff)，確保資料最終一致性。 |
| **P2** | **資源洩漏 (UI/Timeout)** | Medium | 優化視窗回收邏輯與 API 逾時處理，提升長時間運行的穩定度。 |
| **P3** | **分類與抓取邏輯優化** | Low | 重構為動態設定檔與更通用的字串處理邏輯，提升維護性。 |
